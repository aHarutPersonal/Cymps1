Normalize, deduplicate, and compute ages for achievement candidates.

PURPOSE (CMPYS-specific):
Transform raw achievement candidates into a canonical idol timeline suitable for:
- age-based comparison (idol at age X),
- progress scoring,
- grounded chat context,
- plan generation.

INPUT:
- idol_birth_date: "{idol_birth_date}"  // may be null; format "YYYY-MM-DD" if present
- candidates:
{candidates_json}

OUTPUT RULES (MANDATORY):
- Output MUST be valid JSON only (no markdown, no commentary).
- Output MUST match the schema exactly.
- Never invent missing dates, ages, or facts.
- Preserve evidence integrity: do not alter evidence snippets.
- Each output timeline item must have at least 1 evidence item.

Return JSON EXACTLY matching this schema:

{
  "timeline": [
    {
      "canonical_title": "string",
      "canonical_description": "string",
      "date": "YYYY-MM-DD|null",
      "date_precision": "day|month|year|unknown",
      "age_at_event": null,
      "category": "career|learning|finance|impact|mindset|other",
      "importance_score": 0.0,
      "confidence": 0.0,
      "evidence": [
        {
          "source_id": "string",
          "chunk_index": 0,
          "source_url": "string",
          "snippet": "string",
          "confidence": 0.0
        }
      ]
    }
  ],
  "dedupe_notes": [
    {
      "merged_titles": ["string"],
      "reason": "string"
    }
  ]
}

CANONICALIZATION RULES:
- canonical_title: short, action-oriented, unique (e.g., “Founded X”, “Joined Y as Z”, “Acquired Z”, “Published A”).
- canonical_description: compact, neutral public-biography copy in 1 sentence, or 2 only when needed for clarity.
- Lead with the concrete event. Include trajectory or impact only when the candidate evidence supports it; do not manufacture significance merely to complete a template.
- Use active verbs and natural syntax. Vary sentence construction across events without becoming ornate.
- Never include second-person advice, coaching lessons, celebratory hype, or labels such as “Key takeaway”, “Why it mattered”, “Replicable angle”, or “This teaches us”.
- Do NOT add new details that are not supported by candidate descriptions/evidence.
- Keep category consistent with the most appropriate candidate category.

DEDUPE RULES (strict but safe):
Goal: merge true duplicates without accidentally merging distinct events.

You MAY merge candidates if ALL are true:
1) Same action type (founded/joined/acquired/published/launched/etc.) AND
2) Same entity or clearly same event (same company/org/book/role) AND
3) Dates do not conflict (same year OR one is unknown and the other known) AND
4) Descriptions/evidence clearly refer to the same milestone.

You MUST NOT merge if:
- entities differ (different companies/books/roles),
- the action type differs,
- the dates conflict (e.g., 1956 vs 1965),
- or the event could be distinct (e.g., “joined company” vs “became CEO”).

When merging:
- choose the best canonical_title (most specific and widely supported).
- merge evidence: keep up to 3 strongest evidence items across candidates.
- confidence becomes the MAX of merged candidates, then adjusted by rules below.
- add a dedupe_notes entry listing merged_titles and the reason.

AGE COMPUTATION RULES:
- Preserve any source-stated age in the description/evidence, but set
  age_at_event to null in your output. The backend computes and validates age
  deterministically from idol_birth_date, the partial-date convention below,
  and verbatim evidence. Do not perform date arithmetic yourself.

DATE NORMALIZATION RULES (preserve the extraction convention):
- Day precision keeps its exact YYYY-MM-DD value.
- Month precision keeps YYYY-MM-01; day 01 remains a storage sentinel.
- Year precision keeps YYYY-01-01; January 1 remains a storage sentinel.
- Unknown precision keeps date=null.
- Never upgrade precision or infer a missing year/month. date_precision is the
  authority; sentinel components are not factual claims.

IMPORTANCE SCORE RULES (0.0–1.0):
Assign importance_score based on contribution to the idol’s trajectory and usefulness for CMPYS comparison:
- 0.85–1.00: defining milestones (major founding, CEO appointment, flagship product, landmark publication, widely recognized award, major philanthropic foundation)
- 0.60–0.84: significant milestones (major promotion, key partnership, important acquisition, meaningful shift in strategy with documented outcomes)
- 0.30–0.59: supporting milestones (education completion, early roles, smaller launches)
- 0.10–0.29: minor milestones (only if still concrete and evidenced)
Do not inflate importance just because the person is famous.

CONFIDENCE ADJUSTMENT RULES:
Start from the chosen/merged candidate confidence, then adjust:
- If date_precision == "day": no penalty.
- If date_precision == "month": subtract 0.05 (approximate age).
- If date_precision == "year": subtract 0.10 (approximate age).
- If merged candidates were clearly duplicates: add +0.03 (more corroboration).
- If merged candidates were borderline/ambiguous: subtract 0.07.
- If evidence is weakly phrased (“may have”, “reportedly”): cap confidence at 0.69.
Clamp final confidence to [0.0, 1.0].

EVIDENCE RULES:
- Keep evidence snippets verbatim exactly as provided.
- For merged items, prefer evidence that directly supports:
  (a) the action, and (b) the timing.
- evidence.confidence should reflect directness (often ~candidate confidence).
- Keep 1–3 evidence items; never exceed 3.

SORTING RULES:
Sort dated items chronologically by their preserved date value, then put
unknown dates last. For the same stored date, use importance_score descending.
The backend re-sorts by its computed age after validation. Do NOT fabricate
dates or ages to sort.

FINAL SANITY CHECK (MANDATORY):
Before outputting:
- Remove items that are not concrete actions/milestones.
- Remove duplicates not merged.
- Ensure each item has category in allowed set and evidence length >= 1.
- Ensure JSON schema is respected.
- Set age_at_event to null for every item; the backend owns age computation.
