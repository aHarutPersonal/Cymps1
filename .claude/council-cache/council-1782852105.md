## 🔳 Codex (performance) - gpt-5.5
Performance work should start as engineering investigation, not “make things faster” refactoring. The goal is to find the actual bottleneck, fix the root cause, and prove the change improved performance without weakening correctness or maintainability.

**1. Measure First**

Before changing code, define what “slow” means.

Measure:

- User-facing latency: page load, API response time, interaction delay.
- Backend latency: p50, p95, p99 response times.
- Throughput: requests/sec, jobs/minute, messages/sec.
- Resource usage: CPU, memory, DB connections, network, disk I/O.
- Error rate and timeouts.
- Frontend metrics: LCP, INP, CLS, bundle size, render time.

Use profiling tools, not intuition:

```text
Backend:
- APM: Datadog, New Relic, OpenTelemetry, Honeycomb
- Profilers: async-profiler, py-spy, pprof, dotnet-trace
- Logs with request IDs and timing spans
- DB query logs and EXPLAIN plans

Database:
- slow query log
- query planner / EXPLAIN ANALYZE
- index usage stats
- connection pool metrics

Frontend:
- Chrome Performance tab
- Lighthouse
- React Profiler / Vue devtools
- bundle analyzer
- WebPageTest
```

Add lightweight instrumentation around critical paths:

```ts
const start = performance.now();

const result = await loadDashboard(userId);

logger.info("dashboard.load", {
  durationMs: performance.now() - start,
  userId,
});
```

You want evidence like:

```text
Dashboard request p95: 2800ms
DB time: 2100ms
External API time: 300ms
Serialization: 200ms
App logic: 100ms
```

That tells you where to work.

---

**2. Highest-Leverage Areas**

**Database and Queries**

The database is often the biggest source of application slowness.

Look for:

- Missing indexes.
- Queries scanning large tables.
- Over-fetching columns.
- Sorting/pagination without indexes.
- Expensive joins.
- Large OFFSET pagination.
- Lock contention.
- Too many round trips.

Bad:

```sql
SELECT * FROM orders WHERE customer_id = ? ORDER BY created_at DESC;
```

Better:

```sql
SELECT id, total, status, created_at
FROM orders
WHERE customer_id = ?
ORDER BY created_at DESC
LIMIT 50;
```

With an index:

```sql
CREATE INDEX idx_orders_customer_created
ON orders(customer_id, created_at DESC);
```

For large pagination, prefer keyset pagination:

```sql
SELECT id, total, created_at
FROM orders
WHERE customer_id = ?
  AND created_at < ?
ORDER BY created_at DESC
LIMIT 50;
```

---

**N+1 Queries**

N+1s are common and highly damaging.

Bad:

```ts
const orders = await db.orders.findMany();

for (const order of orders) {
  order.customer = await db.customers.findById(order.customerId);
}
```

Better:

```ts
const orders = await db.orders.findMany({
  include: {
    customer: true,
  },
});
```

Or batch explicitly:

```ts
const orders = await getOrders();

const customerIds = [...new Set(orders.map(o => o.customerId))];
const customers = await getCustomersByIds(customerIds);

const customersById = new Map(customers.map(c => [c.id, c]));

return orders.map(order => ({
  ...order,
  customer: customersById.get(order.customerId),
}));
```

Guardrail: add query-count tests for important endpoints.

```ts
expect(queryCounter.count).toBeLessThanOrEqual(5);
```

---

**Caching**

Cache only after confirming repeated expensive work.

Good candidates:

- Read-heavy reference data.
- Expensive computed results.
- External API responses.
- Auth/session/user permission lookups.
- Homepage or dashboard summaries.
- Static or semi-static assets.

Use the right cache level:

```text
Browser cache: static assets, images, JS/CSS
CDN cache: public pages, assets, API responses safe to share
Application cache: computed data, permission maps
Database cache/materialized views: expensive aggregate queries
Request-local cache: avoid duplicate work within one request
```

Example request-local cache:

```ts
class RequestContext {
  private users = new Map<string, Promise<User>>();

  getUser(id: string) {
    if (!this.users.has(id)) {
      this.users.set(id, userRepo.findById(id));
    }

    return this.users.get(id)!;
  }
}
```

Be disciplined about invalidation:

```text
Cache key includes all inputs.
TTL is explicit.
Stale data tolerance is documented.
Invalidation path is tested.
Sensitive/user-specific data is never cached globally.
```

---

**Network and Payload Size**

Slow applications often send too much data too often.

Investigate:

- Large JSON payloads.
- Over-fetching fields.
- Duplicate requests.
- Chatty APIs.
- Missing compression.
- Large images.
- Unpaginated endpoints.
- Waterfall request chains.

Bad:

```json
{
  "user": "...",
  "orders": [10000 items],
  "auditLog": [50000 items]
}
```

Better:

```json
{
  "user": "...",
  "recentOrders": [50 items],
  "auditSummary": {
    "count": 50000,
    "latestEventAt": "..."
  }
}
```

Tactics:

- Paginate large collections.
- Use field selection.
- Compress responses.
- Avoid fetching data not visible on screen.
- Batch requests where appropriate.
- Use CDN for assets.
- Resize images server-side.

---

**Algorithmic Complexity**

Sometimes the bottleneck is plain code.

Look for:

- Nested loops over large collections.
- Repeated filtering/sorting.
- Recomputing the same value.
- Inefficient data structures.
- O(n²) behavior hidden inside helpers.

Bad:

```ts
const result = orders.map(order => ({
  ...order,
  customer: customers.find(c => c.id === order.customerId),
}));
```

This is O(n × m).

Better:

```ts
const customersById = new Map(customers.map(c => [c.id, c]));

const result = orders.map(order => ({
  ...order,
  customer: customersById.get(order.customerId),
}));
```

This is O(n + m).

Guardrail: keep optimized code readable. Prefer clear data structures over clever micro-optimizations.

---

**Concurrency and Parallelism**

Check whether independent work is being done sequentially.

Bad:

```ts
const profile = await getProfile(userId);
const permissions = await getPermissions(userId);
const notifications = await getNotifications(userId);
```

Better:

```ts
const [profile, permissions, notifications] = await Promise.all([
  getProfile(userId),
  getPermissions(userId),
  getNotifications(userId),
]);
```

But avoid uncontrolled concurrency:

```ts
// Risky for thousands of items
await Promise.all(items.map(processItem));
```

Prefer bounded concurrency:

```ts
import pLimit from "p-limit";

const limit = pLimit(10);

await Promise.all(
  items.map(item => limit(() => processItem(item)))
);
```

Also examine:

- Thread pool saturation.
- DB connection pool limits.
- Queue backlogs.
- Lock contention.
- Long transactions.
- Synchronous/blocking work on request paths.

---

**Frontend Rendering and Bundle Size**

Frontend slowness is often caused by too much JavaScript, too much rendering, or too much data.

Investigate:

- Initial bundle size.
- Render waterfalls.
- Expensive components.
- Unnecessary re-renders.
- Large lists without virtualization.
- Heavy client-side computation.
- Unoptimized images/fonts.
- Hydration cost.

Tactics:

```text
- Code split routes and heavy components.
- Lazy-load non-critical features.
- Memoize expensive derived values.
- Virtualize long lists.
- Avoid global state updates that re-render the whole tree.
- Move heavy computation to server or worker.
- Reduce third-party scripts.
```

Example:

```tsx
const filteredItems = useMemo(() => {
  return items.filter(item => item.name.includes(search));
}, [items, search]);
```

For large lists:

```tsx
import { FixedSizeList } from "react-window";

<FixedSizeList
  height={600}
  itemCount={items.length}
  itemSize={40}
  width="100%"
>
  {Row}
</FixedSizeList>
```

**3. Guardrails So Quality Does Not Degrade**

Performance work should raise engineering quality, not lower it.

Use these rules:

```text
No optimization without measurement.
No clever rewrite without tests.
No caching without invalidation strategy.
No concurrency without backpressure.
No query change without correctness checks.
No algorithm change without edge-case coverage.
No frontend optimization without visual/behavioral verification.
```

Protect correctness with tests:

- Unit tests for changed algorithms.
- Integration tests for DB/query behavior.
- Contract tests for API payload changes.
- Regression tests for known slow flows.
- Load/performance tests for critical endpoints.
- Query-count tests for N+1-prone paths.

Protect maintainability:

- Prefer simple, local changes first.
- Keep optimized code explicit.
- Document non-obvious performance decisions.
- Encapsulate caching behind clear interfaces.
- Avoid scattering performance hacks across business logic.
- Delete dead or duplicated paths after optimization.

Example of a maintainable cache wrapper:

```ts
class ProductCache {
  constructor(
    private cache: Cache,
    private productRepo: ProductRepository
  ) {}

  async getProduct(id: string): Promise<Product> {
    const key = `product:${id}`;

    return this.cache.getOrSet(key, {
      ttlSeconds: 300,
      load: () => this.productRepo.findById(id),
    });
  }

  async invalidateProduct(id: string): Promise<void> {
    await this.cache.delete(`product:${id}`);
  }
}
```

This keeps caching policy visible and testable.

**4. Practical Optimization Process**

Use a loop:

```text
1. Define the slow user journey or endpoint.
2. Capture baseline metrics.
3. Profile and identify the largest bottleneck.
4. Form one hypothesis.
5. Make one targeted change.
6. Run correctness tests.
7. Re-measure performance.
8. Keep the change only if it improves the target metric.
9. Document the reason if the optimization is non-obvious.
```

Example:

```text
Problem:
Dashboard p95 latency is 3.2s.

Evidence:
DB time is 2.4s.
Endpoint runs 187 queries.
Most are repeated user permission lookups.

Hypothesis:
Request performs N+1 permission queries.

Fix:
Batch permission loading by user IDs.

Verification:
Query count reduced from 187 to 6.
p95 reduced from 3.2s to 900ms.
Existing integration tests pass.
Added query-count regression test.
```

That is the standard you want.

---

**5. What Usually Pays Off First**

In most applications, investigate in this order:

1. Database query performance and indexes.
2. N+1 queries and excessive DB round trips.
3. Payload size and over-fetching.
4. Missing or poorly designed caching.
5. Sequential work that can safely run concurrently.
6. Algorithmic hotspots.
7. Frontend bundle size and unnecessary rendering.
8. Infrastructure limits: CPU, memory, connection pools, queues.

Micro-optimizations should come late. Improving a loop from 10ms to 2ms does not matter if the request spends 2 seconds waiting on the database.

The best performance improvements are usually boring: fewer queries, smaller payloads, better indexes, less repeated work, bounded concurrency, and measured caching.

## 🟦 Gemini (scalability) - gemini-3.1-pro-preview
As a Scalability Architect, my philosophy is that **performance is a feature, but maintainability is the foundation**. Premature optimization leads to spaghetti code; blind optimization leads to regressions. 

To scale a system for 10x to 100x growth while preserving code quality, we must approach this systematically: **Measure, Optimize, Architect, and Protect.**

Here is your actionable blueprint.

---

### Phase 1: Measure & Profile First (Stop Guessing)
You cannot improve what you cannot measure. Guessing where the bottleneck lies usually results in complex, unreadable code that doesn't actually speed up the application.

1.  **Implement Distributed Tracing:** Use OpenTelemetry, Datadog, or New Relic. You need to see the lifecycle of a request across all services. Look at **p95 and p99 latencies**, not averages.
2.  **Database Query Logging:** Enable slow query logs in your database (e.g., Postgres `log_min_duration_statement`). 
3.  **Application Profiling:** Generate Flame Graphs (using tools like `pprof` for Go, `py-spy` for Python, or `async-profiler` for Java) to identify CPU and memory bottlenecks in the code.
4.  **Establish a Baseline:** Write a load test (using k6 or Gatling) simulating your peak traffic. Run it before and after any change.

---

### Phase 2: High-Leverage Optimizations (The Engineer's Toolkit)
Target the areas that yield the highest ROI with the lowest impact on code complexity.

#### 1. The Database & The N+1 Problem
The database is the bottleneck 80% of the time. 
*   **Fix N+1 Queries:** This happens when an ORM executes one query to get a list of items, and then *N* additional queries to get related data. 
    *   *Bad (N+1):* `users = User.all(); for user in users: print(user.profile.bio)`
    *   *Good (Eager Loading):* `users = User.includes(:profile).all()`
*   **Add Missing Indexes:** Analyze your slow queries using `EXPLAIN ANALYZE`. Ensure foreign keys and frequently filtered columns are indexed. 
*   **Push Compute to the DB:** Don't pull 10,000 rows into application memory to sum a column. Use SQL aggregations (`SUM()`, `GROUP BY`).

#### 2. Caching Strategies (Do this *after* fixing queries)
Do not use caching to mask terrible database performance. Once queries are optimized, introduce caching.
*   **Read-Through/Write-Through Cache:** Use Redis or Memcached for frequently accessed, rarely changing data (e.g., user permissions, configuration).
*   **CDN:** Offload all static assets and cacheable API responses (using `Cache-Control` headers) to Cloudflare or AWS CloudFront.

#### 3. Network & Payload Optimization
*   **Pagination & Sparse Fieldsets:** Never return unbounded lists. Implement cursor-based pagination for large datasets. Allow clients to request only the fields they need (GraphQL is great for this, or use `?fields=id,name` in REST).
*   **Compression:** Ensure gzip or Brotli compression is enabled on your API Gateway/Load Balancer.

#### 4. Concurrency & Algorithmic Complexity
*   **Algorithmic Complexity:** Look for nested loops `O(N^2)` in your business logic. Use HashMaps/Sets `O(1)` for lookups instead of arrays `O(N)`.
*   **Concurrency:** If a single request requires calling three independent downstream APIs, execute them concurrently rather than sequentially.
    ```javascript
    // BAD: Sequential (Takes 3 seconds)
    const user = await getUser(id);
    const orders = await getOrders(id);

    // GOOD: Concurrent (Takes 1 second)
    const [user, orders] = await Promise.all([getUser(id), getOrders(id)]);
    ```

#### 5. Frontend Rendering & Bundle Size
*   **Bundle Splitting & Lazy Loading:** Don't send a 5MB JavaScript payload on the first load. Split bundles by route.
*   **Virtualization:** If rendering large lists/tables, use virtualized lists (e.g., `react-window`) to only render DOM elements currently visible on screen.

---

### Phase 3: Architecting for 10x - 100x Growth
When code-level optimizations are exhausted, you must shift the architecture.

1.  **Horizontal Scaling (Statelessness):** Ensure your application servers are 100% stateless. Move session state to Redis and uploaded files to S3. This allows your auto-scaler to spin up 100 instances flawlessly.
2.  **Asynchronous Processing:** Remove heavy lifting from the synchronous request-response cycle.
    *   *Example:* If a user uploads a video, don't process it in the HTTP request. Save the raw file, publish an `UploadCompleted` event to Kafka/RabbitMQ/SQS, return a `202 Accepted` to the user, and let a background worker process it.
3.  **Database Scaling (CQRS & Sharding):**
    *   *Step 1 (10x):* Implement Read Replicas. Route all `SELECT` queries to replicas, and `INSERT/UPDATE` queries to the primary node.
    *   *Step 2 (100x):* Database Sharding. Partition your data across multiple database instances based on a Shard Key (e.g., `tenant_id` or `user_id`). 

---

### Phase 4: Guardrails for Code Quality
Optimization often creates a tension with readability. Here is how you protect quality:

1.  **Characterization Tests First:** Before refactoring a slow function, ensure it has 100% test coverage for its *behavior*. If you change an `O(N^2)` algorithm to `O(N log N)`, the test suite must prove the output hasn't changed.
2.  **Encapsulate Complexity:** If you must write highly optimized, slightly less readable code (e.g., complex raw SQL instead of a clean ORM call), **hide it behind a clean interface**. 
    ```python
    # The interface remains highly readable and maintainable
    class UserRepository:
        def get_active_users_with_stats(self) -> List[UserStats]:
            # Complex, highly-optimized raw SQL query goes here.
            # The rest of the application doesn't need to know how ugly this is.
            pass
    ```
3.  **Automated Performance Regressions:** Integrate tools like Lighthouse (for frontend) or k6 (for backend) into your CI/CD pipeline. Fail the build if a PR drops performance below your established baseline.
4.  **Feature Flags:** Wrap major architectural optimizations (like switching to a new caching layer or a new async workflow) in feature flags. This allows you to dial up traffic to the optimized path gradually and instantly roll back if error rates spike, without doing a full code revert.
